# 전체적인 이름 리팩토링 (완료)

## 목적

- 솔직히 너무 더럽긴 했음..
- 이게 내가 프로그래밍과 네이밍에 감이 없을 시절에 지은걸 지금까지 쓰고있는거라..
- 그래도 앞으로는 이름을 대충 짓지 말자는 교훈은 얻은 듯.

## 이름 변경 대상

1. `NetState` 구조체 (제일 심각)
2. `PacketType` 열거형 (SAFE 바꿔야함)

## 이름을 바꿔야 하는 이유

### `NetState` 구조체

1. `if_header_error` / `if_error`가 대체 무슨 에러인지 알기 어렵다.
    - 그래서 명확히 `protocol_error` / `transport_error`로 변경해야한다.
    - 명확히 protocol error / transport error가 확실히 연상된다.
2. `if_peer_exit` / `if_peer_error`도 무슨 상태인지 명확하지 않다.
    - 그래서 명확히 `peer_closed` / `peer_protocol_error`로 수정한다.
    - 명확히 상대가 연결을 정상적으로 종료한 경우 / 상대가 유효하지 않은 패킷을 받아서 에러 패킷을 전송하는 경우가 명확히 연상된다.

```cpp
struct NetState {
    bool header_send = false;
    bool payload_send = false;
    bool header_recv = false;
    bool payload_recv = false;

    bool protocol_error = false;
    bool transport_error = false;
    bool peer_closed = false;
    bool peer_protocol_error = false;
};
```

### `PacketType` 열거형

1. `SAFE`만 보고 해당 패킷이 무슨 패킷인지 구별할 수 없음.
    - 초기에 정상 패킷이 한 종류밖에 없을 때는 문제 없었지만,
    닉네임 시스템을 설계하며 새로 다른 정상 패킷 타입이 추가될 때 `SAFE`라는 이름은 확실히 문제가 있음.
    - `CHAT_MESSAGE`로 변경 예정
2. `HEADER_ERROR`는 상관 없을 듯.
    - 변경 안해도 됨.
3. `NICKNAME_CHANGE` 패킷 타입도 닉네임 시스템 적용 전에 임시로 추가.

```cpp
enum class PacketType : int32_t {
    CHAT_MESSAGE = 1,
    NICKNAME_CHANGE = 2,
    HEADER_ERROR = 3
};
```

# 클라이언트 리팩토링 (미완료)

## 원인과 목적

1. 원인 - 닉네임 시스템을 구현하다 어떻게 클라이언트에서
닉네임 설정 / 일반 메시지 패킷을 구분해서 전송할 수 있을지에 대한 문제가 발생
   - 기존에는 단순히 입력받음 = 서버에 일반 메시지 전송, quit 입력받음 = 클라이언트 종료
   - 이렇게 판별하고 있었기 때문에..
   - 구조 개선이 필요했음..
2. 원인 - 애초에 클라이언트 코드가 싱글스레드 에코서버의 클라이언트 코드에서 딱히 수정이 없음
3. 목적 - 클라이언트에서 입력받을 때 각각의 동작을 구별해서 수행하는 기능이 필요함
4. 목적 - 클라이언트 구조를 최신화해야할 필요가 있음

## 정책 설계

### 0. 입력에 따라 동작을 구분

1. 따로 클라이언트에서 받은 입력을 파싱하는 `InputParser` 객체를 둠.
    - `InputParser` 객체의 역할
      - 클라이언트가 받은 입력을 보고 각각의 동작에 대해 보내야 할 `PacketType`을 반환
      - 같이 입력받은 버퍼에 보낼 메시지 저장해줌
      - 각각의 동작은 어차피 서버에 보내야 할 패킷 타입으로 구분할 수 있기 때문에
      - 실제 송신은 담당하지 않음.
2. 그리고 그렇게 반환된 패킷 타입과 채워진 버퍼를 기반으로 서버에 패킷 송신.

이건 [여기서..](./26.06.04%20Inputparser%20구조체%20설계.md) 계속.

### 1. 클라이언트 구조 최신화

1. 로그 출력에 `LineLogger` 객체 적용(현재진행)
2. 해당 클라이언트의 정보를 저장하는 `ClientApp` 객체 설계 및 추가(`InputParser` 객체 설계 및 구현 후)
3. 전체적인 송 / 수신 흐름 재검증 및 개선(이건 언제할지 모르겠다)

#### 1-1. 클라이언트의 정보를 담고있는 객체 필요(2번)

따로 해당 클라이언트의 정보를 담고있는 `ClientApp` 객체를 만들어서

- `ConnectSocket` 캡슐화
- 클라이언트에 대한 모든 조작을 해당 객체로 해결
- `SendPacket()` / `RecvPacket()` 함수로 송 / 수신 함수 추상화

정도의 목적을 달성할 수 있음.

레퍼런스는 서버의 `ClientSession` 객체로.
해당 객체를 본따서 뺼 기능 뺴고 넣을 기능 넣고.

#### 1-2. 로그 출력에 `LineLogger` 객체 적용(1번)

- 클라이언트 전용 로그 출력 인터페이스 함수 필요..
  - 입력받기 전용 함수(기존 함수는 맨 끝에 `\n`이 있어서 자동줄바꿈되어 입력을 정상적으로 받지 못함.)
  - 클라이언트 전용 로그 출력 함수
    - 서버에서 받은 메시지를 출력하는 함수 필요(추후 브로드 캐스트로 확장 시 닉네임도 출력할 수 있는 구조 필요)
    - 메시지 송신 로그는 불필요
    - 클라이언트 전용 에러 로그 함수 필요

```text
[누가 주체로 보낸 메시지인지(즉, NickName)][PacketType] msg
```

NickName은 만약 에코 메시지라면 서버에서 `EchoFromServer`를 닉네임으로 해서 전송, 클라이언트는 그냥 닉네임으로 인식해서 출력
PacketType은 말그대로 수신한 패킷 타입
