# ClientApp 객체

## 목적

- 클라이언트에서 서버의 `ClientSession`처럼
해당 클라이언트의 동작을 추상화
- 그리고 해당 클라이언트가 들고있는 자원을 캡슐화(private 접근제어자로 보호)
- 해당 클라이언트의 동작을 담당하는 인터페이스를 오직 해당 객체만으로 제공

## 설계

> ClientApp = ConnectSocket + NetState + nickname_을 소유하고,
> 사용자 입력 → InputParser → SendPacket/quit/error 처리 → RecvPacket → 출력
> 흐름을 관리하는 클라이언트 최상위 객체

역할 분리는 이런 느낌.

```text
ClientApp
  → 해당 클라이언트의 정보 / 동작을 모아놓은 객체

InputParser
  → 입력을 파싱해서 입력의 의미를 나타내는 ParsedInput을 반환하는 객체

ParsedInput
  → 파싱된 입력, 즉 해당 입력으로 처리해야하는 작업을 나타내는 객체
```

### 해당 객체는 뭘 소유해야할까?

- `ConnectSocket` `sock_` : 이건 필수. 해당 클라이언트가 서버와 통신할 때 사용하는 소켓이니까 필수.
  - 근데 이건 절대 복사로 못 가져오기 때문에 move semantic으로 가져와야 함.
- `NetState` `state_` : 이것도 반필수. 해당 클라이언트의 현재 상태를 기록하는데 좋을 듯.
- `NickName` `nick_` : 이거는 어쩔까. 굳이 해당 클라이언트 내에 해당 클라이언트의 닉네임을 저장할 필요가 있을까? 있을 것 같긴 해.. 일반 보류.

`InputParser::Parse()`는 `static`이라 외부에서도 접근할 수 있으므로 소유하지 않음.

### 해당 객체는 어떤 동작 인터페이스를 가지고 있어야할까?

- `GetState()` : 클라이언트의 지금 당장의 상태를 들고와야할 때
- `NetState SendPacket(const char* msg, uint32_t len, PacketType type)`, 추후 다른 패킷에 대한 Send 함수 제작 시 `SendChatMessagePacket()` : 패킷 송신. 필수. 추후 다른 패킷을 송신할 때도 이 함수가 원형.
  - 상태 기록 및 반환 정책은 `ClientSession`의 `SendPacket()` / `RecvPacket()` 함수와 같음. 
    - 해당 `ClientApp`의 전체적인 상태를 기록하는 `state_`
    - 해당 `SendPacket()` / `RecvPacket()` 송 / 수신의 결과를 기록하는 `send_state` / `recv_state`
- `NetState RecvPacket(char* buf)`, 추후 다른 패킷에 대한 Recv 함수 제작 시 `RecvChatMessagePacket()` : 패킷 수신. 필수. 추후 다른 패킷을 수신할 때도 이 함수가 원형.
  - 여기도 상태 기록 및 반환 정책은 `ClientSession`의 `SendPacket()` / `RecvPacket()` 함수와 같음
- `void Run()` : 에코 서버 단계에서는 `입력 받기` -> `파싱 및 예외 입력 처리` -> `send()` -> `recv()` -> `출력`로 이어지는 루프만 구현하면 됨. 추후에 이 함수는 뺼 수도 있을 듯.
- 파싱은 어떻게 해야할까? 이 객체 내부에서? 아니면 외부에서? 모르겠다.
- 파싱한 결과 처리는 일단 `Run()` 함수 내부에 넣어놓자.
- 출력도 물론 `Run()` 함수 내부에 넣어놓고..

### 서버의 `ClientSession`을 레퍼런스로 잡는다고 할 때, 오직 클라이언트에만 필요한게 뭐가 있을까?

- `Input parsing` : 이게 제일 큰 차이점. 콘솔로 **사용자**에게 입력을 받는다는게 제일 큰 차이지. 
  - 그래서 무조건 받은 입력을 `InputParser` 객체로 `ParsedInput` 형태로 파싱해주는 과정이 필요함.
- `사용자가 종료하는 경우`, 즉 `/quit`를 입력받았을 때의 처리 : 이것도 차이점. 
  - 송 / 수신 루프는 끝내버리고 종료 처리를 시작해야함.
- `사용자의 입력이 유효하지 않은 경우` : 이것도 차이점. 
  - 이거는 에러 메시지 출력 -> 다시 송 / 수신 루프 시작으로 이어버리자.
  - 그리고 이거는 무조건 파싱 쪽에 묶어버리는게 좋을 듯.
